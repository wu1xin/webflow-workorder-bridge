# 群聊表与下游群同步 设计文档

> 编写日期：2026-06-25
> 关联：`docs/plans/2026-06-23-multi-upstream-schema-design.md`（schema v2）、`docs/weflow-对接接口规格说明书（work-order-system侧）.md`（下游契约）、`docs/http-api.md`（WeFlow API）

## 1. 背景与目标

当前 `syncService` 把 WeFlow **所有会话**的消息归一化后入 `queue`，没有任何「哪些群可推送下游」的过滤。本期引入**群聊表**，并把「是否可推送」的裁决权交给**下游（work-order-system）**：

1. 收集 WeFlow 全部**群聊**会话，落本地群聊表 `chat_group`。
2. 把群快照发给下游新增的「同步群聊接口」，下游返回**可接收推送**的群名单。
3. 本地按名单标记，之后**只推送被放行群**的消息。

### 1.1 范围与决策

| 决策点 | 结论 |
|--------|------|
| 平台范围 | 本期只做 weflow；表保留 `channel_id`/`platform` 两列，与 `dedup`/`media_cache`/`channel_state` 多上游约定对齐 |
| 系统语义 | 收敛为**「仅群聊转发」**：单聊（`sessionType=single/other`）一律不入队 |
| 群判定 | `/api/v1/sessions` 返回 `type===2`，或 `username` 以 `@chatroom` 结尾 |
| 推送裁决 | 由**下游**决定（白名单语义）；本地默认 `push_allowed=0`，未经下游确认不推送 |
| 下游接口 | 新增第 5 个端点 `syncGroups`，本期一并写入对接规格 |
| 同步时机 | weflow 连接成功（initial/recovery）全量同步；SSE 实时消息命中未知群时增量同步该群 |
| 闸门位置 | 下沉到会话级：不放行的群连消息都不拉；`processMessage` 入队前再补一道防御性校验 |

## 2. 数据流

```
weflow 连接成功(initial/recovery)
  └─ GET /api/v1/sessions  全量拉会话
       └─ 过滤 group(type===2 或 username 以 @chatroom 结尾)
            └─ upsert 进 chat_group（新群 push_allowed=0, sync_status=pending）
                 └─ POST 下游 syncGroups（群快照）
                      └─ 下游返回 allowed 名单 → 回写 push_allowed / sync_status=synced
                           └─ 拉消息时只处理 push_allowed=1 的群 → 入 queue
SSE 实时消息命中未知群 → 单独对该群跑一次「upsert→syncGroups→标记」增量
```

**默认拒绝**：新群在下游确认前 `push_allowed=0`；下游同步失败时 `push_allowed` 维持原值（保守，不误开），下次连接/SSE 触发重试。

## 3. `chat_group` 表（schema v3）

schema v2 → v3，**纯新增表**（`CREATE TABLE IF NOT EXISTS` + 版本号 +1），无破坏性迁移。

```sql
-- 8. chat_group —— 群聊登记 + 下游推送裁决（仅群聊转发的闸门数据源）
CREATE TABLE IF NOT EXISTS chat_group (
  channel_id      TEXT    NOT NULL,                 -- 连接实例（weflow:default）
  platform        TEXT    NOT NULL,                 -- 平台类型
  conversation_id TEXT    NOT NULL,                 -- 群 ID（xxx@chatroom）= 会话 username
  group_name      TEXT,                             -- 群名（displayName/groupName）
  avatar_url      TEXT,                             -- 群头像（前端展示，可空）
  push_allowed    INTEGER NOT NULL DEFAULT 0,       -- 下游裁决：1 可推送 | 0 不推送
  sync_status     TEXT    NOT NULL DEFAULT 'pending', -- 同步下游状态：pending|synced|failed
  synced_at       INTEGER,                          -- 最近一次成功同步下游的时刻（秒）
  last_error      TEXT,                             -- 最近一次同步失败原因
  first_seen_at   INTEGER NOT NULL,                 -- 首次发现该群的时刻（秒）
  last_seen_at    INTEGER NOT NULL,                 -- 最近一次见到该群（会话/消息）的时刻（秒）
  updated_at      INTEGER NOT NULL,                 -- 行更新时刻（秒）
  PRIMARY KEY (channel_id, conversation_id)
);
CREATE INDEX IF NOT EXISTS idx_chat_group_push ON chat_group(channel_id, push_allowed);
```

**关键语义**：`push_allowed` 是**下游说了算**的字段，本地默认 0；只有进过下游 `allowed` 名单才置 1。入队闸门 = 查 `chat_group.push_allowed`，0 或查不到则丢弃。

## 4. 下游 `syncGroups` 接口契约（新增第 5 个端点）

沿用现有规格约定：`task_white_token` 鉴权、统一响应、**`code==1` 才算成功**。

- **URL**：`POST {base}/extra_server/weflow/syncGroups?task_white_token=...`
- **Content-Type**：`application/json`

**请求体**（群快照，全量或单群增量同结构）：

```json
{
  "agentId": "weflow:default",
  "platform": "weflow",
  "groups": [
    { "sessionId": "xxx@chatroom", "groupName": "项目群",
      "avatarUrl": "https://.../g.jpg", "lastMessageAt": 1738713600 }
  ]
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `agentId` | 是 | 代理/连接实例标识（= `channel_id`） |
| `platform` | 是 | 平台类型（多平台留位） |
| `groups[].sessionId` | 是 | 群 ID（`xxx@chatroom`），下游裁决与回写主键 |
| `groups[].groupName` | 否 | 群名，便于下游人工识别 |
| `groups[].avatarUrl` | 否 | 群头像 |
| `groups[].lastMessageAt` | 否 | 该群最近消息时间（秒） |

**响应**（成功 / 肯定 ACK，**白名单**语义）：

```json
{
  "code": 1, "msg": "success", "time": 1750000000,
  "data": { "allowed": ["xxx@chatroom", "yyy@chatroom"] }
}
```

- `data.allowed`：下游裁定**可接收推送**的群 `sessionId` 集合。
- **本次发了、但不在 `allowed` 里的群 → 一律 `push_allowed=0`**（默认拒绝）。
- 失败（`code!=1`）→ 该批群 `sync_status=failed`、记 `last_error`，`push_allowed` 维持原值；下次触发重试。

**幂等**：快照覆盖式声明，下游每次按 `agentId+allowed` 全量裁定，无需幂等键；代理可随时重发。

> 落地时同步更新 `docs/weflow-对接接口规格说明书（work-order-system侧）.md`：新增 §4.5、接口清单加第 5 行、§8.3 移除「sessionType 取值集合待确认」（已明确 group 判定）。

## 5. 服务端改造点

### 5.1 建表与迁移 — `server/src/db/schema.ts`
- `DDL` 末尾追加 `chat_group` 表 + 索引；`SCHEMA_VERSION` 2 → 3。
- 纯新增，无破坏性迁移：`if (current < 2)` 的 DROP 分支不动，DDL 每次跑自动补建新表，版本号写入沿用既有逻辑。

### 5.2 `ChatGroupStore` — 新增 `server/src/db/chatGroup.ts`
照 `ChannelStateStore` 的预编译语句风格：
- `upsertSeen(channelId, platform, convId, {groupName, avatarUrl, lastMessageAt}, now)`：新行 `push_allowed=0/sync_status=pending/first_seen=last_seen=now`；冲突时只更 `group_name/avatar_url/last_seen_at`，**保留** `push_allowed/sync_status`。
- `markSynced(channelId, sentIds[], allowedIds[], now)`：单事务把 `allowed` 置 1、`sent\allowed` 置 0，`sync_status=synced/synced_at=now`。
- `markSyncFailed(channelId, sentIds[], error, now)`：`sync_status=failed/last_error`，`push_allowed` 不动。
- `isPushAllowed(channelId, convId): boolean`：闸门快查（预编译）。
- `listAllowed/listAll(channelId)`：同步与前端展示用。
- `database.ts` 的 `Db` 聚合加 `readonly chatGroup`。

### 5.3 `GroupSyncService` — 新增 `server/src/sync/groupSyncService.ts`
- `syncAll()`：`listSessions` → 过滤 group → 逐群 `upsertSeen` → 拼快照 → `downstream.syncGroups()` → `markSynced/markSyncFailed`。
- `syncOne(convId, meta)`：SSE 遇未知群时只同步该群。
- 注入下游客户端，便于测试注桩。

### 5.4 下游客户端 — 新增 `server/src/downstream/client.ts`
- `syncGroups(req): Promise<{allowed: string[]}>`，封装 `task_white_token` 鉴权（AES-128-ECB，密钥走 config 线下交付）+ `code===1` 判定。
- 目前**首个**下游出站客户端（forwarder 尚未建），新开 `downstream/` 目录；将来 `receiveMessage/uploadMedia/heartbeat` 归此。

### 5.5 入队闸门 — 改 `server/src/sync/syncService.ts`
- `runFullSync`/`runCompensation`：会话过滤改为「仅 group 且 `chatGroup.isPushAllowed` 为真」才 `pullSession`，单聊与未放行群跳过。
- `processMessage`：入队前补一道 `isPushAllowed` 防御性校验（双保险，SSE 实时通道将来复用同一收口）。
- `onConnected`：先 `await groupSync.syncAll()` 再跑消息同步。

## 6. 测试（vitest，内存库）
- `ChatGroupStore`：`upsertSeen` 不覆盖既有裁决；`markSynced` 白名单语义（sent\allowed 置 0）；`isPushAllowed` 快查。
- 闸门：未放行群的消息被 `processMessage` 拦截、不入 queue。
- `GroupSyncService`：注桩下游，验证 group 过滤、快照拼装、allowed 回写、失败保守不误开。

## 7. 未尽事项 / 留待实现期决策
- **新放行群的历史回灌**：群被新放行后，其放行前的历史消息是否补拉？本期默认「放行后向前转发」，历史回灌作为可选/手动同步，不在本期闸门内自动处理。注意：水位 `last_sync_timestamp` 是 **channel 级全局量**，并非按群独立。因此当群 B 晚于群 A 被放行时，下一轮补偿的起点 = 全局水位，群 B 中「放行后、但早于全局水位」的消息也会被一并跳过——比「仅放行前不补」字面更宽。这是全局水位 + 会话级闸门的设计必然结果；若需精确按群补齐，走手动指定起点同步。
- **SSE 实时入队通道**：当前尚未建（forwarder 待接入）；遇未知群时调用 `groupSync.syncAll(channelId, platform, [session])` 做单群增量，并复用 `processMessage` 闸门收口。
- 下游 AES 密钥/`site key`/base URL 的 config 落地，复用 forwarder 接入时的下游配置。
