# WeFlow 媒体判断口径修正（落库层）— 设计

> 范围：仅落库层。消息转发 / 媒体下载 / 上传 / 重试属未实现的「转发模块」，本次不做。

## 背景与问题

实测上游 `GET /api/v1/messages` 后发现两个问题，动摇了原先「信任上游 `media*` 字段」的判断前提：

1. **抓取层硬 bug**：当前 [`fetchMessagesPage`](../../server/src/weflow/restClient.ts) 只传 `talker/offset/limit/start/end`，**没有** `media=1&image=1&voice=1&video=1&emoji=1`。不带这些参数上游**不返回** `mediaType/mediaFileName/mediaUrl`。
   → 后果：线上所有入库消息 `has_media` 恒为 0、`media_json` 恒为 null，前端「含媒体」筛选实际失效。

2. **`media*` 字段不可靠（最终一致）**：即使带上参数，`media*` 字段也只在 **WeFlow 已把文件下载进本地缓存**（`mediaUrl` 指向 `127.0.0.1:5031/api/v1/media/...`）后才出现。实测同为视频，`localId 63/61` 有字段、`localId 64` 无字段；文件 `localId 60` 也无字段。是**异步时序依赖**，非类型依赖。
   → 后果：用 `media*` 字段判断「是不是媒体」必然漏判未缓存的媒体。

3. **`localType` 才是可靠语义来源**：实测覆盖 `1`=文本、`3`=图片、`43`=视频、`47`=动画表情、`(6<<32)|49`=文件、`(57<<32)|49`=引用。项目里 [`revoke.ts` 的 `isFileMessage`](../../server/src/sync/revoke.ts) 已在用 `localType` 拆包，可复用。

## 核心决策

**「是不是媒体消息」（语义）与「媒体文件是否已就绪可下载」（可用性）是两个不同问题**，当前 `has_media` 把两者混在一起。

- 「是不是媒体」→ 只认 `localType`，可靠、即时。
- 「文件是否就绪」→ 看 `media*` 字段 / `sourceRef`，最终一致，属转发模块职责。

落库层把前者存对、后者解耦记录，为将来转发模块铺路。

## 媒体语义分类

新建纯函数模块 `server/src/weflow/mediaType.ts`，集中 `localType` 拆包与媒体分类，并让 `revoke.ts` 复用同一套拆包原语（消除散落的 `U32 / 49 / 6` 常量重复）。

```ts
export type MediaType = 'image' | 'voice' | 'video' | 'emoji' | 'file'

/** 拆打包 localType：低 32 位=基础类型，高 32 位=appmsg 子类型 */
export function unpackLocalType(lt: number): { base: number, sub: number }

/** 媒体语义分类；非媒体返回 null。只认 localType，不看 media* 字段。 */
export function classifyMedia(msg: WeflowMessage): MediaType | null
```

| localType | 类型 | 媒体? | classifyMedia |
|---|---|---|---|
| 1 | 文本 | 否 | null |
| 3 | 图片 | 是 | `image` |
| 34 | 语音 | 是 | `voice` |
| 43 | 视频 | 是 | `video` |
| 47 | 动画表情 | 是 | `emoji` |
| `(6<<32)\|49` | 文件 | 是 | `file` |
| `(57<<32)\|49` | 引用 | 否 | null |
| 10000 | 系统 | 否 | null |

`revoke.ts` 的 `isFileMessage` 改为基于新模块（等价于 `classifyMedia(msg) === 'file'`，仍保留 `<appattach><fileext>` 正则双保险），撤回窗口逻辑不变。

## 数据模型变化（无需 DB 迁移）

- `has_media`：列不变，仍 `media.length > 0 ? 1 : 0`，但 `media.length` 改由 `classifyMedia` 决定。
- `media_json`：在 `MediaDescriptor` 内**新增 `mediaType` 字段**（写进 JSON，**不动表结构**）。
- `sourceRef`：媒体未就绪时为 `null`——即给转发模块的信号「这是 X 类媒体、文件未到、需回查重试」。

## 适配器改写

`server/src/weflow/adapter.ts` 的 `extractMedia`：

```ts
private extractMedia(externalId, message): MediaDescriptor[] {
  const mediaType = classifyMedia(message)
  if (mediaType === null) return []
  return [{
    mediaKey: externalId ?? 'unknown',   // 稳定：只认 serverId，不再拼 fileName
    mediaType,
    fileName: message.mediaFileName ?? null,
    mime: null,
    size: null,
    sourceRef: message.mediaUrl ?? message.mediaLocalPath ?? null,
  }]
}
```

`mediaKey` 从 `${externalId}:${fileName}` 改为仅 `externalId`：入库时 fileName 常未就绪，拼进去会不稳定 / 撞 `'media'`。一消息至多一媒体的前提下 `externalId` 已足够唯一。跨消息按 md5 去重属转发模块 / `media_cache` 职责，本次不掺和。

## 抓取层改写

`fetchMessagesPage` 固定附加 `media=1&image=1&voice=1&video=1&emoji=1`。仅影响上游是否返回 `media*` 字段，不改变返回条数 / 顺序，对去重、撤回对账无副作用。

## 测试

- 新增 `mediaType.test.ts`：覆盖每个 `localType`（1/3/34/43/47/文件/引用/系统）分类正确。
- 改 `adapter.test.ts`：现有靠 `mediaType:'image'` 触发媒体的用例改为靠 `localType:3`；补「`media*` 缺失但 `localType=43` → `has_media=1`、`sourceRef:null`」。
- `revoke.test.ts`：`isFileMessage` 换实现后回归。

## 待验证 / 已知风险

- **文件类导出参数未知**：文件 `localId 60` 在 `media=1&image=1&voice=1&video=1&emoji=1` 下未返回 `media*` 字段，且参数无 `file` 开关。落库层仍会把它判为 `has_media=1`（语义对），但其 `sourceRef` 能否最终拿到，需在转发模块阶段再测文件类导出参数。不阻塞本次改动。
- **媒体就绪率**：SSE 触发后毫秒级回查，大文件 / 视频此刻往往未缓存，入库时 `sourceRef` 多为 `null`。这是预期行为，由转发模块负责后续回查补齐。

## 不做（YAGNI）

- 媒体下载、`media_cache` 写入、下游上传、未就绪重试 —— 属转发模块。
- `media_json` 不加 `ready` 字段；`sourceRef === null` 即「未就绪」信号，无需冗余列。
