# 已知问题：消息撤回检测无法做到实时

> 日期：2026-06-29　状态：**已接受现状（近实时，~30s 延迟）**　影响模块：WeFlow 撤回检测
> 关联：[撤回检测设计](../plans/2026-06-29-weflow-撤回检测-design.md)、[http-api.md](../http-api.md) §2

## 一句话

撤回检测目前是**周期对账扫描**（默认 30s 一轮），从一条消息被撤回到桥接产出 `message.revoke` 事件，**延迟上界 ≈ 一个扫描间隔**；做不到「撤回瞬间即知」的真实时。

## 现状（当前实现）

- 入队的普通用户消息按类型留撤回看守 `revocable_until`（普通 2min / 文件 3h + grace）。
- `reconcileRevokes()` 每 ~30s 一轮，对仍在窗口内的看守消息回查 v1/messages，发现 `serverId` 已翻成撤回态（localType 10000 + `revokemsg`）→ 入队 `message.revoke`。
- 见 [syncService.ts](../../server/src/sync/syncService.ts) `reconcileRevokes`/`reconcileTalker`，间隔常量 `RECONCILE_INTERVAL_SEC`。

## 根因：延迟的根在上游，不在我们的扫描间隔

桥接只能通过两种方式得知撤回：**①上游主动推送，②我们主动查上游**。

1. **撤回在 WeFlow 侧是 v1/messages 的「原地改写」**：同一 `serverId` 的行 localType 由 1 翻成 10000、content 变 `revokemsg` sysmsg，原文消失。它**不是一条新消息**，因此 SSE 的 `message.new` 流里不会出现。
2. **SSE 的 `message.revoke` 事件实测不可靠**：[http-api.md](../http-api.md) §2 文档上列了 `message.revoke`，但实测对方撤回时 SSE 流里**并不出现**该事件；既往结论是「WeFlow 端需对群**逐个手动**开启才能监听撤回，新建群来不及配、覆盖不可靠」。

结论：上游既不可靠地推、撤回又不走 `message.new`，桥接就只剩「主动查」一条路。**而「只能查」的数据源，客户端再怎么改也变不成实时**——延迟来自「下一次查之前的空窗」，是轮询模型固有的，不是我们间隔设小就能消除的。

## 已评估的方案

| 方案 | 能否真实时 | 结论 / 前提 |
| --- | --- | --- |
| **A. 接 WeFlow `message.revoke` SSE** | ✅ 真实时（零延迟，自带 rawid+原文，还能删掉整套扫描） | **被上游限制卡住**：需逐群手动开、新群覆盖不到。除非 WeFlow 提供「全局/批量开启撤回推送」的开关或接口，否则不可靠。 |
| **B. 放行群单独开推送 + 轮询兜底（混合）** | ✅ 放行群真实时 | 同样取决于 WeFlow 是否暴露「按群开启撤回推送」的 **API**（纯 UI 手动勾则无法自动化）。 |
| **C. 压低轮询延迟（事件触发快速复查）** | ❌ 仅近实时（秒级） | 一条消息进来后在其撤回窗口内对该群快速复查几次，可把延迟从 30s 压到秒级；代价是更多 loopback REST，本质仍是轮询。 |

## 触发「真实时」的唯一条件

WeFlow 能**可靠地**（全局或可经 API 批量）推送 `message.revoke`。这是一个上游侧能力问题，需在目标 WeFlow 版本上实测确认：

> 验证法：在 WeFlow 设置里找「撤回/防撤回推送」类开关并开启 → `curl -N "…/api/v1/push/messages?access_token=…"` 挂住 SSE → 让对方撤回一条 → 看流里是否冒出 `event: message.revoke`。

一旦确认可行，改动很小（扩展 [parseRealtimeTrigger](../../server/src/sync/syncService.ts) 认 `message.revoke`、用 `rawid` 直接 `emitRevoke`），并可整体下线 `revocable_until` 看守 + 周期扫描。

## 影响与当前取舍

- **影响**：撤回事件最坏延迟 ≈ 30s。对「记录/转发撤回」这类语义，秒级到几十秒延迟通常可接受，不影响正确性（只影响时效）。
- **取舍**：在 WeFlow 撤回推送不可靠的前提下，**周期对账是唯一能对所有群（含新群）可靠覆盖**的机制；牺牲的是「瞬时性」。
- **可调**：嫌 30s 慢可调小 `RECONCILE_INTERVAL_SEC`（代价是更多 loopback REST），或按方案 C 做事件触发的快速复查；但都到不了真实时。
